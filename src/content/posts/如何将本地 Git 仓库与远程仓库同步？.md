# 如何将本地 Git 仓库与远程仓库同步？

【核心定义】  
将本地 Git 仓库与远程仓库同步，本质是通过 `push`、`pull`、`fetch` 等命令在本地与远程之间交换提交记录，以保持代码版本一致。

【关键要点】  
1. **推送本地提交**：使用 `git push <远程名> <分支名>` 将本地分支的提交上传到远程仓库，若远程分支不存在则自动创建。  
2. **拉取远程更新**：`git pull` 是 `git fetch`（下载远程更新）与 `git merge`（合并到当前分支）的组合操作，用于获取并整合远程变更。  
3. **建立远程关联**：通过 `git remote add <远程名> <远程URL>` 初次关联远程仓库，后续可使用 `git remote -v` 查看关联信息。  
4. **处理同步冲突**：当本地与远程修改同一文件时，需手动解决冲突后重新提交并推送。  
5. **分支追踪设置**：使用 `git branch --set-upstream-to=<远程名>/<分支名>` 建立本地分支与远程分支的追踪关系，简化后续 `pull/push` 操作。

【深度推导/细节】  
- **冲突场景拆解**：  
  Step 1：开发者 A 在本地修改 `file.txt` 第 10 行并推送至远程。  
  Step 2：开发者 B 在本地修改同一文件同一行，执行 `git pull` 时 Git 检测到冲突，在文件中标记 `<<<<<<< HEAD` 与 `>>>>>>> commit-hash`。  
  Step 3：开发者 B 手动保留所需代码，删除冲突标记后执行 `git add` 与 `git commit` 完成合并提交。  
  Step 4：开发者 B 执行 `git push` 将合并结果同步至远程。  

- **强制推送的风险**：`git push --force` 会覆盖远程历史，仅应在重置本地提交后使用（如 `git rebase` 后），团队协作中推荐使用 `--force-with-lease` 避免覆盖他人提交。

【关联/对比】  
- **`git pull` vs `git fetch + merge`**：`git pull` 自动合并可能引入未预期的冲突，而分步操作（先 `fetch` 再 `merge` 或 `rebase`）更利于审查变更。  
- **集中式 vs 分布式版本控制**：Git 的同步是分布式的，每个本地仓库拥有完整历史，而 SVN 等工具需始终连接中央服务器。  

『面试官追问』  
1. **如何撤销一次已推送的提交？**  
   - 方案 1：`git revert <commit-hash>` 生成反向提交，保留历史记录，适合团队协作。  
   - 方案 2：`git reset --hard <commit-hash>` 本地回退后强制推送（需团队协调），会重写历史。  
2. **`git pull --rebase` 与默认 `git pull` 的区别？**  
   - `git pull --rebase` 将本地提交变基到远程更新之后，形成线性历史，避免多余的合并提交。  
3. **如何同步远程已删除的分支？**  
   - 使用 `git fetch --prune` 清理本地缓存的远程分支引用，再通过 `git branch -d <分支名>` 删除本地对应分支。  

【最佳实践补充】  
- **定期同步**：在开始新功能前执行 `git pull` 减少冲突概率。  
- **分支管理策略**：主分支（如 `main`）保护性推送，开发通过特性分支提交，合并时使用 `--no-ff` 保留分支历史。  
- **钩子自动化**：利用 `pre-push` 钩子运行测试，确保推送代码质量。  

（注：本题不涉及数据结构、线程安全等底层机制，故未展开相关要点；若涉及 Git 内部实现如对象存储、引用日志等，可进一步深入。）
