# Redis 实现分布式锁时可能遇到的问题有哪些？

# Redis 分布式锁实现中的核心问题与解决方案

## 【核心定义】
Redis 分布式锁是通过 Redis 的原子性操作，在分布式系统中实现互斥访问共享资源的一种协调机制，其本质是在 Redis 中设置一个具有特定标识和生命周期的键值对来代表锁的持有状态。

## 【关键要点】
1. **死锁风险**：客户端获取锁后崩溃，锁无法自动释放。解决方案是为锁设置合理的过期时间（TTL）。
2. **误删锁问题**：客户端 A 的锁过期后被 B 获取，此时 A 完成操作并误删 B 的锁。解决方案是使用唯一标识（如 UUID）作为锁的值，删除时验证标识匹配。
3. **锁续期难题**：业务操作时间可能超过锁的过期时间。解决方案是实现“看门狗”机制自动续期，或使用 Redlock 等成熟算法。
4. **集群环境下的可靠性问题**：主从切换可能导致锁状态丢失。解决方案是使用 Redlock 算法在多个独立 Redis 节点上同时获取锁。
5. **性能与可用性平衡**：频繁的锁竞争会影响系统吞吐量。解决方案是合理设计锁粒度、使用非阻塞的尝试获取锁机制。

## 【深度推导/细节】

### 锁误删问题的逻辑拆解
**场景还原**：
- Step 1：客户端 A 获取锁，设置 key=`lock:order`，value=`uuid_a`，TTL=10s
- Step 2：A 的业务操作耗时 12s，第 10s 时锁自动过期
- Step 3：客户端 B 在第 11s 成功获取锁，value=`uuid_b`
- Step 4：A 在第 12s 完成操作，执行 `DEL lock:order` 删除了 B 的锁
- Step 5：客户端 C 此时可以获取锁，导致 A 和 B 同时操作共享资源

**解决方案实现**：
```lua
-- Lua 脚本保证原子性
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### Redlock 算法分步拆解
Redlock 用于解决单点故障问题，要求至少 5 个独立的 Redis 主节点：

Step 1：客户端记录当前时间 T1
Step 2：依次向 N 个节点发送 SET 命令（包含唯一值、过期时间）
Step 3：计算获取锁耗时 = 当前时间 T2 - T1
Step 4：判断成功条件：
- 成功获取锁的节点数 ≥ N/2 + 1
- 总耗时 < 锁的过期时间
Step 5：如果获取失败，向所有节点发送删除命令

## 【关联/对比】

### Redis 分布式锁 vs ZooKeeper 分布式锁
| 维度 | Redis 分布式锁 | ZooKeeper 分布式锁 |
|------|---------------|-------------------|
| 实现原理 | 基于 SETNX 命令 | 基于临时顺序节点 |
| 性能 | 更高（内存操作） | 较低（需要持久化） |
| 可靠性 | 主从切换可能丢锁 | CP 模型，强一致 |
| 死锁处理 | 依赖 TTL 自动过期 | 会话结束自动删除 |
| 惊群效应 | 需要客户端自旋重试 | 通过 Watch 机制避免 |

### Redis 单节点锁 vs Redlock
- **单节点锁**：实现简单、性能高，但存在单点故障风险
- **Redlock**：容错性更强，但实现复杂、性能较低、需要更多资源

## 『面试官追问』

1. **为什么 Redlock 建议使用 5 个节点而不是 3 个？**
   - 5 个节点允许最多 2 个节点故障（N/2+1=3），提供了更好的容错能力
   - 3 个节点只能容忍 1 个节点故障，故障概率相对较高

2. **SETNX 和 SET 命令在实现分布式锁时有什么区别？**
   - SETNX 是原子性的“不存在才设置”，但需要额外命令设置过期时间，不是原子操作
   - Redis 2.6.12+ 的 SET 命令支持 NX 和 EX 参数，可以原子性地完成“设置值+过期时间”

3. **如何避免锁过期时间设置不当的问题？**
   - 采用“看门狗”机制：启动后台线程，在锁过期前定期续期
   - 合理评估业务最长时间，设置缓冲时间（如业务预估 10s，设置 TTL=30s）
   - 使用可重入锁记录持有次数，确保只有最后一次释放才真正删除锁

4. **Redis 集群模式下分布式锁有什么特殊问题？**
   - 异步复制延迟：主节点写入锁后，在复制到从节点前崩溃，可能导致锁丢失
   - Redlock 算法正是为解决此问题而设计，但仍有争议（见 Martin Kleppmann 的分析）

## 【直击痛点：关键数字与设计合理性】

- **锁过期时间 30s**：平衡安全性与性能。太短易在业务未完成时过期，太长则客户端崩溃后资源被长时间锁定
- **重试等待时间 100ms**：避免频繁重试对 Redis 造成压力，同时减少锁获取延迟
- **Redlock 的 5 个节点**：基于容错能力和成本考虑的折中方案（允许 2 个节点故障）
- **时钟偏移容忍度 10ms**：Redlock 要求各节点时钟偏移不超过此值，否则可能影响锁的安全性

## 【版本差异】
- **Redis 2.6.12 前**：使用 SETNX + EXPIRE 组合，非原子操作存在竞态条件
- **Redis 2.6.12+**：支持 SET key value NX EX seconds，原子操作成为标准做法
- **Redis 6.0+**：支持 ACL 和客户端缓存，可增强锁的安全性
- **Redisson 框架**：封装了完整的分布式锁实现，包括可重入锁、公平锁、读写锁等高级特性

## 【最佳实践总结】
1. 优先使用成熟的客户端库（如 Redisson）而非自己实现
2. 为锁设置唯一标识，避免误删
3. 合理设置过期时间，并考虑实现续期机制
4. 在集群环境下评估是否需要 Redlock 或等待 Redis 官方的高可用锁方案
5. 设计降级策略，在分布式锁不可用时保证系统基本可用性
