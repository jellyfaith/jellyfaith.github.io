# MyBatis 中 #{} 和 ${} 的区别

## 【核心定义】
`#{}` 是 MyBatis 的**预编译参数占位符**，本质是 JDBC 的 `PreparedStatement` 参数设置；`${}` 是**字符串替换占位符**，本质是 SQL 语句的字符串拼接。

## 【关键要点】
1. **安全性与 SQL 注入**：
   - `#{}`：**安全**，能有效防止 SQL 注入。MyBatis 会将其解析为 `?`，参数值会经过 JDBC 的类型处理和安全转义。
   - `${}`：**不安全**，存在 SQL 注入风险。MyBatis 会直接将其替换为对应的参数值字符串，不做任何转义。

2. **预编译与执行效率**：
   - `#{}`：**支持预编译**。SQL 语句先被编译为模板，参数后续传入，数据库可对同一模板进行缓存和复用，提升效率。
   - `${}`：**不支持预编译**。每次都会生成一条全新的 SQL 语句，无法利用数据库的预编译缓存。

3. **参数处理与类型转换**：
   - `#{}`：**自动进行类型转换**。MyBatis 会根据 Java 类型和 JDBC 类型自动处理（如日期格式化、字符串转义）。
   - `${}`：**纯粹的字符串替换**。传入什么值就替换为什么字符串，需要开发者自行保证格式正确。

4. **适用场景**：
   - `#{}`：适用于 **WHERE 条件、INSERT/UPDATE 的值**等几乎所有参数传入场景。
   - `${}`：适用于 **动态表名、动态列名、ORDER BY 子句**等 SQL 语句本身结构需要动态变化的场景。

## 【深度推导/细节】
### 核心矛盾：安全性与灵活性的权衡
- **`#{}` 的安全机制**：其底层通过 `PreparedStatement.setXxx()` 方法设置参数。数据库驱动会确保参数值被当作数据而非指令处理，例如引号会被转义。
- **`${}` 的注入风险推导**：
  ```sql
  -- 假设传入参数为: `admin' OR '1'='1`
  SELECT * FROM user WHERE name = '${name}'
  -- 替换后成为:
  SELECT * FROM user WHERE name = 'admin' OR '1'='1'
  -- 这将导致条件永远为真，数据泄露。
  ```
- **`${}` 的合法使用场景逻辑**：
  ```sql
  -- 动态排序，`#{}` 会导致 ORDER BY 'age'，语法错误
  ORDER BY ${orderByColumn}
  -- 动态表名，分表场景
  SELECT * FROM ${tableName}
  ```
  在这些场景中，替换的内容是 **SQL 语法的一部分**，而非数据值。使用时必须确保参数值来源绝对可信（如内部枚举、常量），或经过严格白名单校验。

## 【关联/对比】
- **与 JDBC 的关联**：
  - `#{}` 对应 `PreparedStatement`，是**现代 JDBC 编程的最佳实践**。
  - `${}` 对应 `Statement` 的字符串拼接，是**已被淘汰的不安全做法**。
- **与 ORM 框架的对比**：
  - JPA/Hibernate 使用**命名参数**（如 `:name`），其原理和安全性等同于 MyBatis 的 `#{}`，同样采用预编译。
  - MyBatis 提供 `${}` 是为了在灵活编写原生 SQL 时，保留对 SQL 结构的完全控制力，这是其与全自动 ORM 框架的一个关键设计差异。

## 『面试官追问』
1. **追问一：什么情况下必须使用 `${}`，无法用 `#{}` 替代？**
   - **答案**：当需要动态改变 SQL 的**标识符**（如表名、列名）或 **SQL 关键字部分**（如 ORDER BY 后的列名、GROUP BY 子句）时。因为 `#{}` 会给这些内容加上引号，导致语法错误。

2. **追问二：使用 `${}` 时，如何防范 SQL 注入？**
   - **答案**：必须实施**白名单校验**。例如，对于动态排序字段，预先定义允许的字段集合，传入参数时进行校验。
     ```java
     private static final Set<String> ALLOWED_ORDER_COLUMNS = Set.of("id", "name", "create_time");
     if (!ALLOWED_ORDER_COLUMNS.contains(inputParam)) {
         throw new IllegalArgumentException("Invalid order by column");
     }
     ```

3. **追问三：`#{}` 底层是如何处理复杂对象（如 Map、POJO）的？**
   - **答案**：通过 OGNL 表达式解析。例如 `#{user.name}`，MyBatis 会调用 `user.getName()` 方法获取值。对于 Map，则调用 `map.get("key")`。

4. **追问四：模糊查询 `LIKE` 应该用 `#{}` 还是 `${}`？如何正确编写？**
   - **答案**：必须使用 `#{}`。正确写法是在 Java 代码中拼接好通配符，或将拼接放在 SQL 中。
     ```sql
     -- 写法1：参数中带通配符
     WHERE name LIKE #{pattern}  -- pattern = "%张%"
     -- 写法2：SQL中使用CONCAT（数据库相关）
     WHERE name LIKE CONCAT('%', #{keyword}, '%')
     ```
     绝对禁止 `LIKE '%${value}%'`，这是高危的注入点。

## 【版本差异与最佳实践】
- **版本一致性**：`#{}` 和 `${}` 的行为在 MyBatis 3.x 所有版本中保持一致。
- **最佳实践**：
  1. **默认永远使用 `#{}`**。
  2. 使用 `${}` 时，必须**审查参数来源**，进行严格的输入校验或白名单过滤。
  3. 在 XML 映射文件中，对 `${}` 的使用添加明显的**注释警告**，说明其风险和使用原因。
  4. 可考虑使用 MyBatis 的 `@Provider` 注解或动态 SQL 标签（如 `<if>`、`<choose>`）来替代部分需要使用 `${}` 的动态场景，以提升安全性。
