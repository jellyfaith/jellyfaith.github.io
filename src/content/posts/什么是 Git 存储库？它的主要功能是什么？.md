# 什么是 Git 存储库？它的主要功能是什么？

【核心定义】  
Git 存储库（Repository）是一个由 Git 版本控制系统管理的目录，其中包含了项目的所有文件、历史提交记录、分支、标签等元数据，本质是一个**内容寻址的文件系统**，通过哈希值唯一标识和追踪内容变更。

【关键要点】  
1. **数据结构**：Git 存储库的核心由四种对象构成——**Blob**（存储文件内容）、**Tree**（存储目录结构）、**Commit**（存储提交信息与父子关系）、**Tag**（标记特定提交）。所有对象均通过 SHA-1 哈希值寻址。  
2. **核心算法**：采用 **快照式存储** 而非差异存储。每次提交会为所有被追踪文件创建新快照，未修改的文件通过指针复用历史对象，确保高效存储与完整历史追溯。  
3. **版本控制流程**：通过 **工作区 → 暂存区（Index）→ 本地仓库** 的三级结构实现精确的版本管理。暂存区作为中间层，允许用户选择性提交修改。  
4. **分支机制**：分支本质是**指向某个提交的可移动指针**（如 `master`、`feature`）。创建与切换分支成本极低，支持非线性协作开发。  
5. **分布式架构**：每个开发者拥有完整的本地仓库，可独立提交历史，再通过 **推送（push）** 与 **拉取（pull）** 与远程仓库同步，实现去中心化协作。

【深度推导/细节】  
- **冲突解决逻辑**：  
  Step 1：当多人修改同一文件同一区域时，Git 无法自动合并，会标记为“冲突状态”。  
  Step 2：用户需手动编辑文件，保留所需内容，并删除冲突标记（`<<<<<<<`、`=======`、`>>>>>>>`）。  
  Step 3：将解决后的文件重新加入暂存区并提交，生成新的合并提交节点。  
- **扩容与性能优化**：  
  - Git 对大型仓库采用 **包文件（packfile）** 压缩机制，将相似对象进行增量编码，显著减少存储空间。  
  - 引用日志（reflog）记录所有分支指针变动，允许误操作后恢复历史状态，相当于“操作保险机制”。

【关联/对比】  
- **Git vs. SVN（集中式版本控制系统）**：  
  - Git 为分布式，每个本地仓库完整独立；SVN 依赖中央服务器，单点故障风险高。  
  - Git 提交本地化，支持离线工作；SVN 必须联网提交。  
  - Git 分支轻量且操作快；SVN 分支是目录拷贝，操作笨重。  
- **Git 存储库 vs. 工作目录**：  
  存储库（`.git/` 目录）存储所有元数据与历史；工作目录是用户可见的文件编辑区域，内容从存储库检出。

『面试官追问』  
1. **Git 如何保证数据的完整性？**  
   - 所有对象均通过 SHA-1 哈希生成唯一标识，任何内容修改都会改变哈希值，历史记录不可篡改。  
   - 对象间通过哈希指针链接，形成有向无环图（DAG），确保依赖关系可验证。  
2. **如果提交时漏了文件，如何修正？**  
   - 使用 `git commit --amend` 可修改最近一次提交（添加漏文件或重写提交信息）。  
   - 若需修改更早提交，需使用交互式变基（`git rebase -i`），但需注意已推送历史的强制推送风险。  
3. **Git 如何高效存储大量相似文件？**  
   - 通过 **对象去重**：相同内容的文件仅存储一个 Blob 对象。  
   - 定期自动执行 `git gc`（垃圾回收），将松散对象打包为 packfile，使用增量压缩算法减少冗余。

【线程安全与版本差异】  
- **线程安全**：Git 设计为单线程操作工具，通过文件锁（如 `index.lock`）保证并发操作时仓库状态的一致性。多线程场景需由上层封装协调。  
- **版本差异**：  
  - Git 2.20+ 开始实验性支持 SHA-256 哈希算法，以应对 SHA-1 的碰撞风险。  
  - Git 2.23 引入 `git switch` 与 `git restore` 命令，分离分支切换与文件恢复功能，提升操作语义清晰度。

（注：Git 存储库本身不直接涉及“0.75、8、64”等数字型临界点设计，此类参数常见于 HashMap 等数据结构。Git 的核心参数如哈希长度（SHA-1 为 40 位十六进制）、默认分支名（`master` 或 `main`）属于约定型设计。）
